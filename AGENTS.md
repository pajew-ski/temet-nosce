# AGENTS.md

Public GitHub repo, project name **temet-nosce** (means "know thyself" or "know yourself"). Content: an n8n agent that explains itself and its own blueprint through graph retrieval. The graph describes the repo it lives in, and the path for building such a system from scratch, including this AGENTS.md as the first component.

Target audience: complete beginners to graphs. The goal isn't production readiness, it's that someone comes away thinking in nodes and edges instead of folders and prose.

## Non-Goals

- No vector DB, no embedding step
- No build tool, no framework, no bundler for the website
- No auth (public showcase)
- No manual dark/light toggle. Automatic only, via `prefers-color-scheme`

## Data Model

### Node Schema (`graph/nodes/<id>.md`)

```yaml
---
id: <string, == filename without .md>
type: Project | Concept | Component | Step | Limitation
edges:
  - rel: <string>
    target: <id of another node>
---
<free text body, currently notes only>
```

Edges are directed, defined once. No back-edges. The index is globally visible to the agent, so direction only matters for maintenance, not for retrieval.

### Index Schema (`graph/index.md`)

```markdown
## <id>
type: <type>
notes: <comma separated>
edges: <rel> -> <target>, <rel> -> <target>
path: graph/nodes/<id>.md
```

No longer hand-maintained. Generated by `scripts/sync-graph.mjs` from the node frontmatters. One block per node, flat text, fits entirely into the system prompt of the n8n AI Agent node.

## Sync Script (`scripts/sync-graph.mjs`)

- Bun, no external dependencies. The frontmatter schema is constrained enough for a small hand-written parser, no need for `js-yaml` or similar
- Reads all `graph/nodes/*.md`, extracts `id`, `type`, `notes`, `edges`
- Writes `graph/index.md` per the index schema
- Writes `docs/graph-data.json` as `{nodes: [{id, type}], edges: [{source, target, rel}]}`
- Validates along the way: every edge must point to an existing `id`, otherwise it errors with file name and line, non-zero exit code
- Invocation: `bun scripts/sync-graph.mjs`, no flags needed

## GitHub Action (`.github/workflows/sync-graph.yml`)

- Trigger: push to `graph/nodes/**`
- Steps: checkout, install Bun via `oven-sh/setup-bun` (GitHub-hosted runners don't ship Bun preinstalled the way they do Node, this is the one third-party action needed), run `bun scripts/sync-graph.mjs`, commit and push if `graph/index.md` or `docs/graph-data.json` changed (plain git CLI beyond that)
- Purpose: editing a node file is enough, the index and the visualization catch up automatically. A forgotten local sync gets caught by the action.

## Design System

Beauty is a first-class requirement, not a finishing touch. Every spacing, sizing, and typography decision derives from a single constant: φ (1.618).

**Color**: strictly achromatic, no hue, light and dark share the same scale

```css
:root {
  color-scheme: light dark;
  --bg: oklch(98% 0 0);
  --surface: oklch(94% 0 0);
  --border: oklch(85% 0 0);
  --text: oklch(15% 0 0);
  --text-muted: oklch(40% 0 0);
}
@media (prefers-color-scheme: dark) {
  :root {
    --bg: oklch(15% 0 0);
    --surface: oklch(20% 0 0);
    --border: oklch(30% 0 0);
    --text: oklch(95% 0 0);
    --text-muted: oklch(70% 0 0);
  }
}
```

`oklch(L 0 0)`: chroma is 0, hue is meaningless at chroma 0, so the palette cannot tint. Lightness steps are chosen for contrast, not for φ. Forcing a ratio onto perceptual lightness would fight accessibility rather than serve it. φ belongs in the geometry, not in the gray values.

**Spacing**: Fibonacci sequence, converges to φ, and is the same sequence nature uses for phyllotaxis and shell growth. One ladder, reused for padding, gaps, margins, and border-radius.

```css
:root {
  --space-1: 5px;
  --space-2: 8px;
  --space-3: 13px;
  --space-4: 21px;
  --space-5: 34px;
  --space-6: 55px;
  --space-7: 89px;
  --space-8: 144px;
  --radius-sm: var(--space-1);
  --radius-md: var(--space-2);
  --radius-lg: var(--space-3);
}
```

**Typography**: base size stays at the browser default (`1rem`, respects user zoom and accessibility settings), scale ratio is φ

```css
:root {
  --text-sm: 0.875rem;     /* 14px, captions — deliberately not 1/φ, legibility floor */
  --text-base: 1rem;       /* 16px */
  --text-lg: 1.618rem;     /* φ¹, ~26px, h3 */
  --text-xl: 2.618rem;     /* φ², ~42px, h2 */
  --text-2xl: 4.236rem;    /* φ³, ~68px, h1 / hero */
}
body { line-height: 1.618; }
```

**Layout**: graph canvas and detail side panel split at the golden ratio, not 50/50

```css
.layout { grid-template-columns: 1.618fr 1fr; }
```

The graph canvas itself is a golden rectangle: `aspect-ratio: 1.618 / 1`.

Tailwind is used for layout and spacing utilities only (flex, grid, gap). No Tailwind color classes, the default palette would reintroduce a hue. All color comes from the custom properties above.

## Website (`docs/`)

Stack, both via CDN, no npm, no bundler:

- **Tailwind CSS** (CDN script), layout and spacing utilities only, no color classes (see Design System)
- **Cytoscape.js** (CDN script) for the graph visualization, `cose` layout as default, node/edge colors pulled from the design tokens instead of Cytoscape's defaults

`index.html`, single page, three sections:

1. Short explanation (max. 150 words): what a knowledge graph is, index as retriever instead of similarity function, distinction from vector RAG
2. Graph visualization: fetch `graph-data.json`, render with Cytoscape as a golden rectangle, clicking a node shows its notes and edges in a side panel sized at the φ:1 split
3. n8n chat widget: `@n8n/chat`, fully wired at build time, not a placeholder

   ```html
   <link href="https://cdn.jsdelivr.net/npm/@n8n/chat/dist/style.css" rel="stylesheet" />
   <script type="module">
     import { createChat } from 'https://cdn.jsdelivr.net/npm/@n8n/chat/dist/chat.bundle.es.js';
     createChat({ webhookUrl: 'WEBHOOK_URL' });
   </script>
   ```

   `WEBHOOK_URL` as a single constant at the top of `chat.js`, filled in after importing one of the two n8n workflow variants. Theme the widget by overriding its CSS custom properties with the site's design tokens, for example:

   ```css
   :root {
     --chat--color-primary: var(--text);
     --chat--color-secondary: var(--text-muted);
     --chat--color-white: var(--bg);
     --chat--color-light: var(--surface);
     --chat--color-dark: var(--text);
     --chat--spacing: var(--space-4);
     --chat--border-radius: var(--radius-sm);
     --chat--font-family: inherit;
   }
   ```

   Variable names have shifted across `@n8n/chat` versions, confirm the exact names against the installed version before wiring this up. Either way, the effect is the same: the widget loses its default pink/teal accent and inherits the neutral palette.

### Dark / Light Mode

Automatic only, no toggle. Same neutral tokens from the Design System, swapped via `prefers-color-scheme`.

- The `:root` custom properties (`--bg`, `--surface`, `--text`, ...) switch through the `@media (prefers-color-scheme: dark)` block, covers all plain HTML/CSS, including Tailwind's layout utilities
- Cytoscape (canvas) and the chat widget (its own internal styling) don't pick up `:root` changes on their own and need to read the current theme in JS. `docs/theme.js` is one small shared module, no dependency:

  ```js
  export function getTheme() {
    return window.matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light';
  }
  export function onThemeChange(callback) {
    window.matchMedia('(prefers-color-scheme: dark)').addEventListener('change', e => {
      callback(e.matches ? 'dark' : 'light');
    });
  }
  ```

  `graph.js` uses it to pick Cytoscape's style values on init and re-style on change. `chat.js` uses it to re-set the `--chat--*` custom properties on change.

`graph.js` and `chat.js` both stay thin: fetch/init, read theme via `theme.js`, one or two event handlers. No state management, no framework.

## n8n Workflows (`n8n/`)

Two complete, independently runnable variants, same agent, different retrieval tool. Shows that the pattern isn't tied to a specific tool.

**Variant A — `workflow-variant-a-http-tool.json`**
- Chat Trigger node (public, provides the webhook URL for the website widget)
- AI Agent node, system prompt = full content of `graph/index.md`
- Tool: HTTP Request node, GET against `https://api.github.com/repos/<owner>/<repo>/contents/<path>`, header `Accept: application/vnd.github.raw`, one parameter `path`
- Credential: GitHub PAT with no scopes, only needed for a higher rate limit

**Variant B — `workflow-variant-b-github-node.json`**
- same Chat Trigger + AI Agent node
- Tool: native n8n GitHub node, resource `File`, operation `Get`, parameters `Owner`/`Repository`/`File Path` taken from the `path` field of an index block
- Credential: GitHub App or OAuth2 credential through n8n's own node type

Tool description is identical in both cases: the path parameter comes from the `path` field of an index block.

`n8n/README.md`: setup steps for both variants, credential setup, note that the webhook URL needs to be entered in `docs/chat.js` after import, short comparison of when to use which variant (A transfers to other APIs, B needs less configuration for a GitHub-only use case).

## Build Order

1. Create the 20 node files under `graph/nodes/`
2. Write `scripts/sync-graph.mjs`
3. Run the sync script to generate `graph/index.md` and `docs/graph-data.json`
4. Build `docs/style.css` (design tokens per the Design System section) and `docs/theme.js`
5. Build `docs/index.html`, `graph.js`, and `chat.js` (widget fully integrated and themed, not a placeholder)
6. Build `n8n/workflow-variant-a-http-tool.json` and `n8n/workflow-variant-b-github-node.json`, write `n8n/README.md`
7. Set up `.github/workflows/sync-graph.yml`
8. Write the root `README.md` (short, link to the GitHub Pages URL)
